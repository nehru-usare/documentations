## 1Ô∏è‚É£ Observability Overview

Observability in Java means **collecting and correlating three key pillars**:

| Type | Tools | Purpose |
|------|--------|----------|
| **Metrics** | Micrometer, Prometheus, Grafana | Numeric KPIs (heap, latency, throughput) |
| **Logs** | Logback, Loki, ELK | Structured event history |
| **Traces** | OpenTelemetry, Jaeger | Request flow visualization |

‚úÖ Good observability = clear insight into system behavior without changing code.  
‚úÖ JVM-level observability adds another dimension ‚Äî **GC, JIT, memory, and threads**.

## 2Ô∏è‚É£ JFR (Java Flight Recorder)

**Java Flight Recorder (JFR)** is a low-overhead profiler built into the JDK.

### üîπ Start JFR Recording:
```bash
java -XX:StartFlightRecording=name=app,filename=app.jfr,duration=60s,settings=profile MyApp
````

### üîπ Continuous Recording:

```bash
java -XX:StartFlightRecording=filename=recording.jfr,maxage=1h,maxsize=250M,dumponexit=true MyApp
```

### üîπ Analyze Results:

Open `app.jfr` in **Java Mission Control**.

### JFR Captures:

* GC events
* Memory allocations
* Thread states
* Locks and synchronization
* CPU samples
* Class loading metrics

‚úÖ Safe for production ‚Äî overhead < 1‚Äì2%.
‚úÖ Perfect for capturing intermittent spikes or leaks.

## 3Ô∏è‚É£ JMC (Java Mission Control)

**Java Mission Control (JMC)** is the official GUI for analyzing JFR recordings.

### üîπ Installation:

```bash
sudo apt install jmc
```

### üîπ Key Tabs:

* **Memory** ‚Üí Heap usage, GC, allocation hotspots
* **Threads** ‚Üí Contention, deadlocks
* **I/O** ‚Üí File and socket metrics
* **Code** ‚Üí Method-level hot paths
* **Events** ‚Üí GC pauses, safepoints, exceptions

> üí° Combine JMC + JFR for continuous application performance insight.

## 4Ô∏è‚É£ Async Profiler

**Async Profiler** is a low-level native sampling profiler for JVMs.

### üîπ Download:

[https://github.com/jvm-profiling-tools/async-profiler](https://github.com/jvm-profiling-tools/async-profiler)

### üîπ Run Profiling:

```bash
./profiler.sh -d 30 -e cpu -f profile.html <pid>
```

### üîπ Generate Flame Graph:

```bash
./profiler.sh -d 30 -e cpu -o flamegraph -f flame.html <pid>
```

‚úÖ Captures CPU and allocation samples.
‚úÖ Works even inside containers.
‚úÖ Produces Flame Graphs for visualization.

> üß† Flame Graphs reveal *where* your app spends time ‚Äî ideal for micro-optimizations.

## 5Ô∏è‚É£ Micrometer Metrics Integration

**Micrometer** is the de facto metrics facade for JVM and Spring Boot.

### üîπ Maven Dependency:

```xml
<dependency>
  <groupId>io.micrometer</groupId>
  <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
```

### üîπ Basic JVM Metrics:

```java
MeterRegistry registry = new SimpleMeterRegistry();
new ClassLoaderMetrics().bindTo(registry);
new JvmMemoryMetrics().bindTo(registry);
new ProcessorMetrics().bindTo(registry);
```

### üîπ Spring Boot Integration:

Enable `/actuator/prometheus` endpoint:

```yaml
management:
  endpoints:
    web:
      exposure:
        include: "prometheus"
```

‚úÖ Micrometer auto-collects:

* Heap & non-heap usage
* GC time
* CPU load
* Thread states

## 7Ô∏è‚É£ OpenTelemetry Tracing

**OpenTelemetry (OTel)** provides distributed tracing and metric correlation across microservices.

### üîπ Enable Tracing Agent:

```bash
java -javaagent:opentelemetry-javaagent.jar \
     -Dotel.service.name=order-service \
     -Dotel.exporter.otlp.endpoint=http://otel-collector:4317 \
     -jar app.jar
```

### üîπ Spring Boot Configuration:

```yaml
management:
  tracing:
    sampling:
      probability: 1.0
```

‚úÖ Compatible with:

* **Jaeger**
* **Grafana Tempo**
* **AWS X-Ray**
* **Elastic APM**

‚úÖ Automatically tracks:

* HTTP requests
* Database calls
* Messaging events

> üß† Add traceId + spanId in logs for full request reconstruction.

## 11Ô∏è‚É£ Common JVM Bottlenecks and Fixes

| Issue                   | Cause                      | Fix                            |
| ----------------------- | -------------------------- | ------------------------------ |
| **High GC pauses**      | Excessive allocation       | Tune G1GC, reduce object churn |
| **CPU saturation**      | Thread contention          | Use Virtual Threads            |
| **Slow startup**        | Class loading, JIT warm-up | Use GraalVM Native Image       |
| **Memory leaks**        | Unreleased collections     | Use JFR heap dump analysis     |
| **Deadlocks**           | Synchronized blocks        | Monitor with `jstack`          |
| **Blocked I/O threads** | Blocking calls             | Adopt async or virtual threads |
