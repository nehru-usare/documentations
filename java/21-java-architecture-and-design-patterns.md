## 1Ô∏è‚É£ Architectural Overview

Modern enterprise Java systems typically follow:
- **Layered architecture** ‚Üí clear separation between layers
- **Hexagonal/Clean architecture** ‚Üí independence from frameworks
- **Domain-driven design** ‚Üí business logic at the core
- **Microservice patterns** ‚Üí modular and autonomous services

‚úÖ **Goals of good architecture:**
- Maintainability  
- Testability  
- Scalability  
- Separation of concerns  
- Technology independence

## 2Ô∏è‚É£ Layered Architecture

The **traditional architecture** most enterprise apps start with.

### Layers:
1. **Presentation Layer (Web/UI)**  
   - Handles REST controllers or UI interactions.
2. **Service Layer (Business Logic)**  
   - Encapsulates domain logic and rules.
3. **Repository Layer (Persistence)**  
   - Handles database operations.
4. **Domain Layer (Entities + Value Objects)**  
   - Represents core business data.

```text
Controller ‚Üí Service ‚Üí Repository ‚Üí Database
````

‚úÖ Easy to understand
‚úÖ Works well for CRUD apps
‚ùå Harder to scale for complex domains

### Example:

```java
@RestController
@RequestMapping("/users")
class UserController {
    private final UserService service;
    UserController(UserService service) { this.service = service; }

    @GetMapping("/{id}")
    public ResponseEntity<UserDto> getUser(@PathVariable String id) {
        return ResponseEntity.ok(service.getUser(id));
    }
}
```

## 3Ô∏è‚É£ Hexagonal (Ports & Adapters) Architecture

Introduced by **Alistair Cockburn**, this architecture isolates **core logic** from **external systems**.

### üîπ Key Concepts:

* **Ports** ‚Äî Interfaces for input/output boundaries
* **Adapters** ‚Äî Implement those interfaces for technologies (DB, API, UI)
* **Domain** ‚Äî The pure business logic

```
         +-----------------------+
         |   Application Core    |
         |  (Ports + Domain)     |
         +-----------+-----------+
                     |
        +------------+------------+
        | Adapters: REST, DB, MQ  |
        +-------------------------+
```

‚úÖ Technology-agnostic
‚úÖ Highly testable
‚úÖ Easier refactoring (swap DB or API)

### Example:

```java
interface UserRepositoryPort {
    User findById(String id);
}

class JpaUserRepositoryAdapter implements UserRepositoryPort {
    private final UserJpaRepository repo;
    public User findById(String id) {
        return repo.findById(id).orElseThrow();
    }
}
```

## 4Ô∏è‚É£ Clean Architecture Principles

Promoted by **Robert C. Martin (Uncle Bob)**.

* **Entities (Core domain)** ‚Äì independent business logic
* **Use cases** ‚Äì interact with domain logic
* **Interface adapters** ‚Äì convert between domain and framework
* **Frameworks/drivers** ‚Äì external tools (Spring, DB, Web)

```text
Frameworks ‚Üí Adapters ‚Üí Use Cases ‚Üí Entities
```

‚úÖ Framework independence
‚úÖ Testable use cases
‚úÖ Business rules don‚Äôt depend on infrastructure

> üí° Core rule: *‚ÄúDependencies always point inward.‚Äù*

## 5Ô∏è‚É£ Domain-Driven Design (DDD)

DDD focuses on modeling **complex business domains** using **ubiquitous language**.

### Building Blocks:

* **Entity** ‚Äî Identity + Lifecycle
* **Value Object** ‚Äî Immutable, no identity
* **Aggregate** ‚Äî Cluster of entities treated as one
* **Repository** ‚Äî Retrieve aggregates
* **Service** ‚Äî Domain operation

### Example:

```java
record Money(BigDecimal amount, String currency) {}
class Account {
    private final String id;
    private Money balance;

    void deposit(Money money) {
        balance = new Money(balance.amount().add(money.amount()), balance.currency());
    }
}
```

> üß© Use DDD for **complex**, business-heavy systems, not CRUD APIs.

## 7Ô∏è‚É£ Modularization and Java Platform Module System (JPMS)

Introduced in Java 9, modules enforce boundaries inside monoliths.

### Example: `module-info.java`

```java
module com.nehrudev.user {
    exports com.nehrudev.user.api;
    requires com.nehrudev.common;
}
```

‚úÖ Enforces encapsulation
‚úÖ Improves maintainability and security
‚úÖ Enables better startup and footprint control

> üí° Use JPMS for large codebases to avoid cyclic dependencies.

## 8Ô∏è‚É£ Creational Design Patterns

| Pattern              | Purpose                     | Java 21 Example      |
| -------------------- | --------------------------- | -------------------- |
| **Singleton**        | One instance only           | Enum-based singleton |
| **Factory Method**   | Centralized object creation | Static factory       |
| **Builder**          | Stepwise construction       | Fluent Builder       |
| **Prototype**        | Clone existing objects      | `clone()`            |
| **Abstract Factory** | Family of related objects   | Dependency injection |

### Example (Builder Pattern):

```java
record User(String name, int age) {
    static class Builder {
        private String name;
        private int age;
        Builder name(String n) { this.name = n; return this; }
        Builder age(int a) { this.age = a; return this; }
        User build() { return new User(name, age); }
    }
}
```

## 9Ô∏è‚É£ Structural Design Patterns

| Pattern       | Use Case                               | Example            |
| ------------- | -------------------------------------- | ------------------ |
| **Adapter**   | Bridge between incompatible interfaces | Legacy integration |
| **Decorator** | Add features dynamically               | Logging decorator  |
| **Composite** | Hierarchical structures                | Filesystem nodes   |
| **Facade**    | Simplify subsystem access              | Service Facade     |
| **Proxy**     | Control access                         | Security proxy     |

### Example (Decorator Pattern):

```java
interface Notifier { void send(String message); }

class EmailNotifier implements Notifier {
    public void send(String message) { System.out.println("Email: " + message); }
}

class SlackNotifier implements Notifier {
    private final Notifier notifier;
    SlackNotifier(Notifier n) { this.notifier = n; }
    public void send(String message) {
        notifier.send(message);
        System.out.println("Slack: " + message);
    }
}
```

## üîü Behavioral Design Patterns

| Pattern                     | Description                   |
| --------------------------- | ----------------------------- |
| **Strategy**                | Select algorithm dynamically  |
| **Observer**                | Event-driven communication    |
| **Command**                 | Encapsulate requests          |
| **Template Method**         | Skeleton algorithm with steps |
| **Chain of Responsibility** | Pass request through handlers |

### Example (Strategy Pattern):

```java
interface PaymentStrategy { void pay(double amount); }
class CreditCardPayment implements PaymentStrategy {
    public void pay(double amount) { System.out.println("Paid via Card: " + amount); }
}
class UpiPayment implements PaymentStrategy {
    public void pay(double amount) { System.out.println("Paid via UPI: " + amount); }
}
class PaymentProcessor {
    private final PaymentStrategy strategy;
    PaymentProcessor(PaymentStrategy s) { this.strategy = s; }
    void process(double amount) { strategy.pay(amount); }
}
```

## 11Ô∏è‚É£ Reactive and Asynchronous Design Patterns

‚úÖ Use **CompletableFuture**, **Virtual Threads**, or **Project Reactor** for async flows.
‚úÖ Avoid blocking calls; prefer non-blocking I/O.
‚úÖ Implement backpressure and circuit breakers.

Example using **CompletableFuture**:

```java
CompletableFuture.supplyAsync(() -> service.fetchData())
    .thenApply(data -> process(data))
    .thenAccept(System.out::println);
```

‚úÖ Combine with **Structured Concurrency** for better control in Java 21.

## 12Ô∏è‚É£ Best Practices Summary

‚úÖ Keep business logic framework-agnostic
‚úÖ Use **sealed interfaces + records** for clean domains
‚úÖ Avoid God classes and circular dependencies
‚úÖ Make services stateless where possible
‚úÖ Prefer **dependency injection (DI)** over singletons
‚úÖ Always **log, monitor, and profile** before optimizing
‚úÖ Use **architectural fitness functions** in CI/CD pipelines

> üß† Simplicity scales better than cleverness.
