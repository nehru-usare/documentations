## 1Ô∏è‚É£ Why Performance Optimization Matters

In enterprise applications, **poor performance = business loss**.  

Optimizing Java applications ensures:
- **Lower latency** for API responses  
- **Better scalability** under load  
- **Reduced cloud cost** and footprint  
- **Higher throughput and concurrency**

> üß© Goal: ‚ÄúDo more with less‚Äù ‚Äî fewer CPU cycles, smaller memory footprint, faster GC.

## 2Ô∏è‚É£ Performance Mindset for Developers

‚úÖ Always **measure first**, never guess  
‚úÖ Profile real workloads, not synthetic tests  
‚úÖ Optimize based on **bottlenecks**, not code size

## 3Ô∏è‚É£ Microbenchmarking with JMH (Java Microbenchmark Harness)

[JMH (OpenJDK)](https://openjdk.org/projects/code-tools/jmh/) is the official framework for **accurate microbenchmarks**.

### Example:
```java
import org.openjdk.jmh.annotations.*;

@State(Scope.Benchmark)
public class StringConcatBenchmark {

    @Benchmark
    public String testPlusOperator() {
        return "Hello " + "World";
    }

    @Benchmark
    public String testStringBuilder() {
        return new StringBuilder("Hello ").append("World").toString();
    }
}
````

### Run Benchmark:

```bash
java -jar target/benchmarks.jar
```

‚úÖ JMH handles **JIT warm-up**, **GC effects**, and **CPU jitter**.
‚úÖ Use for algorithm comparison, serialization tests, caching efficiency, etc.

## 4Ô∏è‚É£ Data Structure Selection and Memory Efficiency

| Goal                   | Recommended Data Structure                  |
| ---------------------- | ------------------------------------------- |
| Fast lookup            | `HashMap`, `ConcurrentHashMap`              |
| Order + fast traversal | `ArrayList`                                 |
| Frequent insert/delete | `LinkedList` (use carefully)                |
| Sorted elements        | `TreeMap`, `TreeSet`                        |
| Thread-safe list       | `CopyOnWriteArrayList`                      |
| High throughput queues | `ArrayBlockingQueue`, `LinkedBlockingQueue` |

> üí° Choose **ArrayList over LinkedList** unless random insert/delete dominates.

### Example:

```java
List<String> list = new ArrayList<>(1000); // Pre-size to avoid resizing
```

## 5Ô∏è‚É£ String, Object, and Record Optimization

### üîπ Use StringBuilder for concatenation in loops:

```java
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) sb.append(i);
```

### üîπ Prefer Records for DTOs:

```java
public record UserDTO(String name, int age) {}
```

‚úÖ Immutable
‚úÖ Compact memory layout
‚úÖ Faster serialization

### üîπ Avoid unnecessary object creation:

* Use `String.intern()` cautiously.
* Reuse frequently used objects.
* Prefer **primitive streams** (`IntStream`, `DoubleStream`) for numeric processing.

## 6Ô∏è‚É£ Concurrency and Parallelism Optimization

### üîπ Virtual Threads for I/O-bound tasks

```java
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 10).forEach(i ->
        executor.submit(() -> {
            Thread.sleep(500);
            return i;
        })
    );
}
```

‚úÖ Millions of lightweight threads
‚úÖ No blocking overhead
‚úÖ Best for microservices

---

### üîπ CompletableFuture for Asynchronous Pipelining

```java
CompletableFuture
    .supplyAsync(() -> service.fetchData())
    .thenApply(data -> process(data))
    .thenAccept(System.out::println);
```

‚úÖ Non-blocking pipelines
‚úÖ Thread reusability

## 7Ô∏è‚É£ I/O and Network Performance

| Task                 | Optimization                                           |
| -------------------- | ------------------------------------------------------ |
| File I/O             | Use `Files.newBufferedReader()` or NIO2                |
| Network I/O          | Use async libraries (Netty, HTTPClient)                |
| JSON serialization   | Use Jackson with `ObjectReader` / `ObjectWriter` reuse |
| Database connections | Use Connection Pooling (HikariCP)                      |
| Caching              | Use in-memory caches (Caffeine, EHCache, Redis)        |

### Example: Efficient Buffered File Reading

```java
try (BufferedReader reader = Files.newBufferedReader(Path.of("data.txt"))) {
    reader.lines().forEach(System.out::println);
}
```

> üí° Always close resources or use try-with-resources.

## 8Ô∏è‚É£ JIT and AOT Compilation Tuning

JIT (Just-In-Time) compiles hot methods into native code.
Java 21 supports **Graal JIT** and **AOT** for faster startup.

### Tuning Flags:

```bash
-XX:+TieredCompilation
-XX:+UseG1GC
-XX:+PrintCompilation
-XX:+UnlockExperimentalVMOptions -XX:+UseJVMCICompiler
```

### Ahead-of-Time Compilation:

```bash
jaotc --output libApp.so MyApp.jar
```

‚úÖ Ideal for microservices and CLI tools with startup constraints.

## üîü Practical Performance Patterns

‚úÖ **Cache frequently used computations**
‚úÖ **Batch operations** instead of frequent small calls
‚úÖ **Avoid reflection** in performance-critical paths
‚úÖ **Reduce GC pressure** by reusing immutable objects
‚úÖ **Profile before optimizing**
‚úÖ **Measure memory footprint** using `-Xlog:gc*`

### Example: Caching Pattern with Caffeine

```java
Cache<String, User> cache = Caffeine.newBuilder()
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .maximumSize(10_000)
    .build();

User user = cache.get("nehru", id -> userService.loadUser(id));
```

‚úÖ Lightweight and high-performance alternative to EHCache.
